# 安伴成果物提交文档修订设计

## 目标

在不改变现有成果物定位的前提下，把 `product/` 目录中的两份 Word
文档修订到可提交状态，并复核现有演示视频：

- `商业计划书_安伴.docx`：覆盖问题、市场、产品、技术与 AI、创新、竞品、商业模式、财务和团队，且所有当前能力与代码一致。
- `product-intro.docx`：详细说明安伴的产品定位、功能、使用方式、软硬件架构、运行部署和验证状态，并提供源码与线上访问地址。
- `demo.mp4`：保留现有真实线上只读演示，仅复核可播放性和内容覆盖，不在本次重录。

## 证据边界

内容以当前工作树、完整 AnBan 文档仓和线上只读状态为准。已读取当前仓库全部第一方代码、文档和有价值的 Git 忽略文件；缓存、编译产物和密钥只确认存在，不展开内容。线上 childweb 的 `index.html`、`app.js`、`sw.js` 与本地文件哈希一致，后端 `/health` 返回 200。

演示阶段不发送留言、不触发问候、不触发设备摄像头，避免对真实设备和现场人员产生外部影响。视频只使用登录、导航和读取现有数据等只读操作。

## 修订原则

- 保留两份文档的既有章节顺序、品牌色和主要版式，采用可追踪的局部改写，不整体推倒重做。
- 当前事实以代码、测试、部署状态和完整文档仓为准；旧 PRD 目标和未来规划必须显式标注，不能写成已实现能力。
- 不写入 token、访问码、绑定码、私钥或环境密钥。
- 所有市场规模、人口、政策和监管结论必须附可追溯来源；未经证实的绝对化表述删除或降级为假设。
- 财务预测必须标注为测算模型，并公开关键假设，不把假设写成已实现收入。

## 商业计划书修订设计

保留现有 Word 商业计划书结构，重点完成以下修订：

1. 将技术架构统一改为方案 C：AnBan 是独立第三服务，xiaozhi 是冻结上游；设备连接、音频流、ASR、核心对话 LLM 和 TTS 由 xiaozhi 链路承担，AnBan 只经 Manager OpenAPI 驱动设备。
2. 以当前实现重写产品能力：账户与设备绑定、管理员/成员角色、状态与时间线、留言、提醒、问候计划、家庭画像、专属记忆、视觉原图历史与重分析、AI 画像、AnBan Mind、PWA 和 Android APK。
3. 将视觉隐私说明改为真实的数据闭环：原图和分析结果会按配置留存，受认证和角色权限保护，并支持重分析、删除和容量控制；生产环境自动视觉轮询默认关闭，手动“看一眼”独立可用。
4. 将记忆说明改为“本地事实记忆 + 对话沉淀 + OpenMemory 兼容读取 + 四层上下文”，删除已实现向量检索等不实表述。
5. 增补当前系统架构图和产品界面图，修复“以下为系统架构示意”但没有图片的问题。
6. 新增“数据来源与参考资料”，至少覆盖人口、养老政策、AI 拟人化互动治理和文中采用的市场规模口径。
7. 扩充财务模型，补充三年情景测算、硬件成本、模型调用成本、获客成本、续费率、售后/退换和现金流敏感项；所有数字均标记假设来源。
8. 删除“行业首个”“绝对领先”等缺少证据的绝对化措辞，竞品结论改为可验证的能力维度比较。
9. 完成“核心团队”章节。若没有实名履历材料，则使用不虚构个人经历的“团队能力与岗位分工”版本；若用户提供真实姓名和背景，则替换为实名版本。

## 产品说明书修订设计

文档采用 A4 纵向的产品说明书形态，主色使用安伴暖橙，辅以深蓝和浅灰。首页是简洁封面；正文使用稳定的标题层级、页眉页脚、真实截图、架构图、短表格和提示框。

保留现有 15 页暖橙产品说明书形态，并进行以下局部修订：

1. 将 childweb 测试数从 59/59 更新为当前实测的 60/60，并保留 web 80/80、Go 全包测试通过。
2. 将 `web/` 标为“旧版兼容前端”，明确当前产品前端为 `childweb/`。
3. 补充账户注册/登录、设备绑定、管理员/成员权限、时间线、语音输入、问候计划、视觉历史/重分析/删除、PWA 和 APK。
4. 完善本地运行说明，写明 `.env.example`、Manager Base URL、API Token、访问码等配置类别，但不展示敏感值。
5. 说明 HTTP 线上地址的 PWA 安装边界，并将 APK 作为移动端推荐入口。
6. 将错误的“图 1”表格题注改为“表 1”，为所有图片补充有意义的替代文本。
7. 更新文档元数据，清除模板残留作者、旧时间和无意义属性。

## 演示视频复核

现有视频保持 1920x1080、16:9、H.264/AAC MP4。它承担产品讲解和线上界面操作记录；公网地址与 APK 作为独立的可运行证据。由于本次没有真实硬件拍摄条件，不虚构或合成“真人操作硬件”镜头，也不把现有视频描述成软硬一体实录。

## 验收

- 商业计划书完整覆盖成果要求中的问题、市场、产品/服务、技术与 AI、创新、竞品和商业模式，并且不再包含已确认的技术事实错误。
- 产品说明书包含产品介绍、功能说明、操作说明、源码地址、线上地址、APK 和可复现运行前提。
- 两份 DOCX 均生成每页 PNG，并逐页确认无裁切、重叠、乱码、破表和异常空白页。
- 两份 DOCX 均无修订痕迹、批注残留、占位文本和无意义模板元数据；图片具有替代文本。
- MP4 为可播放的 H.264/AAC 文件，1920x1080，时长落在目标范围。
- 视频包含真实线上页面操作、产品功能说明、源码地址和线上地址。
- 三个成果物均位于 `product/`，不包含密钥或临时文件。

## 非目标

- 不修改业务代码或线上配置。
- 不触发真实设备写操作。
- 不虚构团队成员、履历、客户、营收、融资或未经证实的市场数据。
- 不把生成脚本、渲染 PNG 或中间视频作为提交成果物。
- 不提交或覆盖当前工作树里已有的 childweb 修改和二进制。
