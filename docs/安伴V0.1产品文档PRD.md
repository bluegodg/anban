# 安伴 V0.1 产品文档（PRD）

> PRD = Product Requirements Document，产品需求文档。在动手前把"要做成什么、为谁做、做哪些不做哪些"写清楚，让自己和任何来帮忙的人（包括 AI 助手）都对齐同一个目标。
> 版本：V0.1（路演 Demo / 产品原型）
> 目标日期：2026-06-16 前可路演
> 更新日期：2026-05-28（按 docs/specs/2026-05-28-prd-plan-refinement-design.md 重写）

---

## 0. 一句话定义

**安伴是一台放在家里的主动式 AI 陪伴设备：它既能和老人自然说话，也能把子女的关心带到老人身边，并能根据时间和环境主动开口。**

V0.1 的工程本质：**把"豆包式云端实时语音 AI"接到一块 ESP32 硬件上，再在服务端加上安伴独有的"主动陪伴 + 子女连接 + 记忆 + 摄像头触发"。**

---

## 1. V0.1 要证明什么（路演目标）

V0.1 不是产品、不量产、不盈利。它只需要让观众相信三件事：

1. 老人能和这只"宠物"自然地说话（语音陪伴成立）。
2. 它会主动关心人，不是傻等人叫（主动陪伴成立 —— 安伴区别于普通音箱的灵魂）。
3. 子女能远程连接到它（留言、看状态 —— 三元关系成立）。

**路演现场必须出现的三个"高光瞬间"（与 §3.1 必演功能编号对齐）：**

| 高光 | 内容 | 对应 §3.1 必演功能 |
|---|---|---|
| ① | 老人对设备说话，它自然回应（最好能边说边打断） | #1 被动语音对话 |
| ② | 设备主动开口问候老人（定时 / 子女按钮 / 提醒 / 视觉触发 任一） | #2 #6 #7 任一 |
| ③ | 子女在手机网页上留言 → 设备过一会儿温柔地把留言播给老人 | #3 子女端留言 |

→ §6 验收标准会把这三条标为"**绝对必须现场**"，不能用兜底视频代替。

## 2. 用户与角色

| 角色 | 是谁 | 在 V0.1 里关心什么 | 路演中扮演什么 |
|---|---|---|---|
| 老人 | 居家、可能独居的长辈 | 有人陪着说话、被主动关心、能听到子女的声音/留言 | 演示者扮演（"王阿姨"） |

#### #1 被动语音对话（含实时打断）

**① 目标**

让老人不用学习任何按键操作，开口说话设备就能听懂并自然回应；说话过程中可随时打断 AI 当前播报（像和家人讲话一样）。这是路演高光①"老人能和宠物自然说话"的硬件载体，也是其他所有必演功能依赖的基础语音管道。

**② 路演高光时刻**

- 演示者对设备说："嗨，你今天看起来真精神～"
- 设备 1 秒内开口回应（例如："哎呀王阿姨您也是，眼睛真亮~"）
- AI 正在说"……今天天气很好，您要不要出去……"时，演示者插话"哎对了，"
- AI 立即停止当前播报，听新一轮输入，回应得当
- 时长约 30 秒

**③ 可量化验收**

- 唤醒到首响应（ASR 完成 → TTS 首 byte）端到端 ≤ 2 秒
- 打断响应（用户开口 → AI 停止播报）≤ 500ms
- 一轮对话内 ASR/LLM/TTS 任一失败 ≤ 5% 频率（10 轮内至少 9 轮成功）
- 连续 20 轮对话不需要重启设备/后端
- 老人语速正常区间能识别（手动测三档：慢速 / 正常 / 略快）

**④ 接口轮廓**

```
设备 ↔ 后端（WebSocket，xiaozhi 协议）
device → server: { type: "audio_chunk", data: bytes, sequence }
device → server: { type: "stop" }   # 表示一轮说完
server → device: { type: "tts_chunk", data: bytes, sequence }
server → device: { type: "interrupt" }  # 收到设备新一轮 audio 后下发，中断当前 TTS

# 内部链路
设备 audio → 后端 → 云端 ASR → 云端 LLM（含记忆/画像注入）→ 云端 TTS → 后端 → 设备
```

#### #2 主动问候（定时 + 子女远程触发）

**① 目标**

设备能根据预设时间（如每天上午 8 点）自动开口问候老人，或由子女在 Web 端点一下按钮立即触发一句。这是路演高光②"它会主动关心人"的最小内核——证明设备不只是被动等待，而是有"惦记"的能力。

**② 路演高光时刻**

- 演示者展示子女端 Web 页面，点击"问候妈妈"按钮
- 设备 5 秒内开口："王阿姨，下午好~ 今天精神咋样？"
- 老人答"挺好的"→ 进入正常对话循环（衔接 #1）
- 或：现场把设备时钟调到 17:00（晚问候时间），设备到点主动说："王阿姨，傍晚啦，吃饭了没？"

**③ 可量化验收**

- 子女端点击按钮到设备开口播报 ≤ 5 秒
- 定时问候到点触发误差 ≤ 30 秒（普通 cron 精度即可）
- 每天预设 3 个时间段（早 / 午 / 晚），可在子女端配置
- 同一 10 分钟窗口至多 1 条主动语音输出（与 #6 提醒共配额，避免抢话）
- 老人正在和 AI 对话时，主动问候应排队等待，不强行打断

**④ 接口轮廓**

```
POST /api/greetings/trigger        子女端按钮立即触发
  body: { deviceId, tonePreset?: "warm"|"casual" }

GET/PUT /api/greetings/schedule    子女端配置定时
  body: { deviceId, slots: [{ time: "08:00", enabled: true }, ...] }

# 复用 #6 的下发消息
{ type: "active_speak", text, contextHints: {scene: "morning_greeting"} }
```

#### #3 子女端留言 → 设备播报

**① 目标**

子女在 Web 端文字输入一条留言（最长 100 字），设备在合适时机用 AI 的声音温柔地播给老人。这是路演高光③"子女远程连接成立"的核心——观众必须看见"子女发的字 → 设备说出的话"这条端到端链路。

**② 路演高光时刻**

- 子女在 Web 端输入："妈，我下班路过老张家了，他说让你有空过去喝茶。"
- 点击"发送"
- 设备 30 秒内开口（最好夹一句衔接："哎刚才小明发来个消息哈～"）后念出留言
- 子女端那条留言状态变 ✅ 已播报

**③ 可量化验收**

- 子女端"发送"到设备播报完毕端到端 ≤ 60 秒
- 留言文字长度 ≤ 100 字（超出截断 + 子女端提示）
- 老人正在对话时排队，不强行打断
- 留言完成播报后子女端 30 秒内可见状态更新
- 单条留言失败不会卡死后续留言（队列健壮）

**④ 接口轮廓**

```
POST /api/messages              子女端发送
  body: { deviceId, text, fromName? }
  resp: { messageId, status: "pending"|"played"|"failed", queuedAt }

GET /api/messages?deviceId=&status=    列表

# WebSocket：子女端订阅状态变化
event: message.played | message.failed
```

#### #4 设备状态

**① 目标**

让子女在 Web 端能看到三件事：① 设备此刻在不在线 ② 老人最近一次和设备说话是什么时候 ③ 自己刚发的留言播了没。这是子女端的"情绪压舱石"——三句话告诉子女"妈还好"。

**② 路演高光时刻**

不是独立场景，而是贯穿 #2 #3 的背景信息——子女打开 Web 页面，顶部即显示："设备在线 🟢" + "最近互动：3 分钟前" + 留言列表里某条标着 ✅ 已播报。

**③ 可量化验收**

- 设备掉线后子女端 ≤ 30 秒内显示"离线 🔴"
- "最近互动时间"准确度 ≤ 1 分钟误差
- 留言状态从"pending → played"延迟 ≤ 10 秒
- 后端重启后状态信息不丢
- 同时打开 2 个浏览器都能看到一致状态（多端同步）

**④ 接口轮廓**

```
GET /api/device/status?deviceId=
  resp: {
    online: bool,
    lastSeenAt,
    lastInteractionAt,
    messages: [{ messageId, status, queuedAt, playedAt? }]
  }

# WebSocket 推送状态变化
event: device.online | device.offline | message.played | interaction.happened
```

#### #5 持久化记忆 + 家庭画像注入

**① 目标**

让 AI 跨会话记住和老人交互过程中沉淀下的信息：① 子女初始录入的家庭画像（称呼、子女/孙辈名字、喜好、作息、健康背景）② 老人在对话中说过的事（"昨天去公园了""我喜欢豫剧"）。每次对话起手前注入到 LLM 系统提示词，让 AI 像家人不像客服。具体实现路线（A 复用 / B 补齐 / C 自建）在 W1 周五决策，见 §5 与 [记忆模块决策文档](./decisions/)。

**② 路演高光时刻**

跨两轮对话演示。第一轮老人问"我孙子叫啥" → AI 答出名字（来自子女端预录画像）。第二轮（10 分钟后）老人说"今天腰有点酸" → AI 自然回应并在下一轮带"您刚才说腰酸，要不要早点休息？"（来自当前会话沉淀）。

**③ 可量化验收**

- 家庭画像 ≥ 8 字段（姓名、称呼、子女/孙辈姓名、喜好、作息、健康背景、忌口）
- 路演至少 2 次成功召回：1 次画像、1 次当前会话沉淀
- 单次 LLM 调用注入的记忆 token ≤ 1500
- 后端重启后画像 + 沉淀记忆不丢
- 子女端 Web 能增删改画像字段

> **降级规则**：若实际运行落到 Level 1 以下（见 §5 / §7），上面"≥ 2 次召回"自动从必演降为加分。

**④ 接口轮廓**

```
GET /api/profile?deviceId=
POST/PUT /api/profile      （子女端编辑）
  body: { fields: { name, nickname, children:[...], grandchildren:[...],
                    hobbies, schedule, health, taboos } }

# 内部：记忆库读写（具体实现随 §5 路线 A/B/C 不同）
memory.recall(deviceId, query, topK) → ranked memories
memory.add(deviceId, content, source: "dialogue"|"profile", ts)
```

#### #6 主动提醒（用药 / 生日 / 节日）

**① 目标**

让"主动陪伴"从"定时说一句"升级成"携带具体内容"——"该吃降压药了""明天是孙子生日"。这是把高光②"它会主动关心人"从抽象口号落到老人**真实记得住的服务**。和 #2 主动问候共享同一条下发管道，工作量增量小。

**② 路演高光时刻**

- 上台前：子女端 Web 添加"今天 14:00 提醒妈妈测血压，要带孙子小宝的口吻"
- 路演中：现场再加一条 90 秒后触发的（避免等真到点）
- 90 秒后设备主动开口："王阿姨，该测血压啦~ 小宝昨天还问起您有没有按时测呢。"
- 老人答"好的"→ 子女端那条提醒变 ✅ 已完成

**③ 可量化验收**

- 子女端创建提醒 → 设备落到本地调度，端到端 ≤ 60 秒
- 播报文本长度 30–60 字（短于 30 干瘪、长于 60 老人记不住）
- 老人语音回"好/知道了/收到" → 状态自动转"已完成"；30 分钟无应答 → 转"未应答"且子女端可见
- 同一 10 分钟窗口至多 1 条主动语音输出（提醒和 #2 #7 共配额）
- 后端重启后已排入提醒不丢

**④ 接口轮廓**

```
POST   /api/reminders                       子女端创建
  body: { deviceId, scheduledAt, content,
          category: med|birthday|festival|custom }
  resp: { reminderId, status: "scheduled" }

GET    /api/reminders?deviceId=&status=    列表/筛选
DELETE /api/reminders/:id                  撤销

# 内部：后端 → 设备 websocket
{ type: "active_speak", text, contextHints: {name, occasion}, reminderId }

# 内部：设备 → 后端
{ type: "reminder_ack", reminderId, ackKind: "voice"|"timeout" }
```

#### #7 轻量视觉触发

**① 目标**

摄像头判定"画面里有人 / 没人 / 有动作"等粗粒度信号，设备由此触发一句主动问候。把"主动陪伴"从"看时钟说话"升级成"看到老人回来才说话"——更有人味。**风险高**：摄像头链路 3 周内稳定度不确定，做不出来按 §7 风险表降级到加分层。

**② 路演高光时刻**

- 老人离开镜头 → 1 分钟后回来坐下
- 摄像头检测到"画面里出现人"
- 5 秒内设备开口："呀王阿姨，您回来啦~ 刚才屋里挺安静的呢。"
- 这一条允许走兜底视频（见 §7）。

**③ 可量化验收**

- 摄像头采帧间隔 ≤ 10 秒（不需要实时）
- "有人 → 无人 → 有人"状态变化触发 1 次问候
- VLM 调用延迟 + 触发延迟 ≤ 8 秒
- 同一 10 分钟窗口至多 1 次视觉触发（与 #2 #6 共配额）

> **降级规则**：若 W2 末视觉链路不稳，视觉触发整条降级到加分层（§3.2），必演减为 6 条。见 §7。

**④ 接口轮廓**

```
# 设备 → 后端
{ type: "frame", data: jpegBytes, ts }

# 后端 → VLM（云端）
POST {vlm_endpoint} { image, prompt: "Is there a person in this image?" }

# 后端内部状态机
PresenceState: NO_ONE → SOMEONE → NO_ONE → SOMEONE   # 转 SOMEONE 时触发问候
```

#### 加 #1 娱乐内容（点播音乐 / 新闻 / 讲故事）

**① 目标**：老人陪伴场景里开机率最高的用途之一。子女在 Web 端预设几个频道（地方戏曲 / 央广新闻 / 评书等），老人说"放段戏听听"就播。

**② 兜底/降级**：用云 API 现成内容源接一两个；不做的时候这条直接砍，路演不提。

---

#### 加 #2 紧急轻量版

**① 目标**：① 子女端有"呼叫妈妈"按钮，按下后设备立即开口"妈，宝宝想您了，您方便吗？"② N 天（默认 3 天）无任何互动 → 后端给子女端推送提醒"妈这几天没说话哦"。不做跌倒检测、不做真实电话。

**② 兜底/降级**：本质是 #3 留言的"加急模式" + #4 状态的"沉默告警"，复用现有管道即可。做不出来路演不提。

---

#### 加 #3 子女端深度能力

**① 目标**：① 历史对话摘要（LLM 把当天对话总结成 100 字给子女）② 音频回放（选几条关键对话能听原音）。

**② 兜底/降级**：摘要复用现有 LLM 调用，回放需要把对话录音上传保存——做不出来直接砍。

---

#### 加 #4 杂项加分包

把 4 个零散加分一行说完：
- "看一眼"截图：子女端点按钮，设备拍一张照传回 Web
- 视觉问答：老人问"看看我手里啥"，拍照 → 云端 VLM 回答
- 方言一两句：接现成方言 ASR/TTS API 演示
- 知识库 RAG：放家庭资料/用药说明给 AI 检索后回答

**② 兜底/降级**：每个都是"接个 API 演一两次"级别。能做哪个上哪个，做不动整包跳过。

---

#### 加 #5 AI 视频通话（类豆包 / Gemini Live）

**① 目标**：老人 ↔ AI 的视频实时对话——摄像头持续上行帧到云端 VLM，AI 边看边聊（不只是"看一眼"，是"一直看着"）。把现有视觉触发 + 看一眼 + 视觉问答升级成连续视频流。

**② 兜底/降级**：

- 工程实现走"设备每 N 秒采一帧 + 云端多模态滚动推理"，不是真 30fps 流（成本/带宽顶不住）
- 与 §3.3 路线图"实时视频监控"边界要清楚：本条是用户主动发起、有明确会话；不是常驻监控
- 数据策略（什么时候采、采完做啥、要不要存）在 §8 讲
- 做不出来直接砍，本期不演

### 3.3 路线图未来（写进 PPT"未来规划"，本期不做）

按"做不了 / 暂不做 / 边界谨慎"三类列出：

**做不了——硬件/能力门槛过高**：
- 智能项链形态（独立电池/穿戴形态需要重新硬件设计）
- 跌倒检测（需要 IMU + 训练 + 安全工程）
- 精准定位（需要 GPS/基站定位授权）
- 复杂安防识别（火灾、入侵、烟雾等）
- 医疗级告警（需医疗认证）

**暂不做——3 周内吃不动**：
- 模型个性化（细粒度声音/性格定制，需要语音克隆 + 微调）
- 子女 ↔ 老人 视频呼叫（人对人，需 WebRTC 信令栈 + 接听 UI）
- 真实电话呼叫
- 原生 App 上架
- 多家庭多账号体系
- 本地大模型部署

**边界谨慎——和现有功能形似但本期不做**：
- 实时视频监控 / 长时间录像（区别于加 #5 AI 视频通话：监控是常驻、无会话；通话是用户主动发起）
- 完整方言体系（只演一两句够）

> PPT 谈"未来规划"时按上述三类讲，避免观众把"暂不做"听成"不会做"。

## 4. 系统架构（V0.1）

```
┌─────────────┐   WiFi/WebSocket   ┌──────────────────────────┐
│  安伴设备     │ ←───────────────→ │  后端 (云服务器 server A)   │
│ 立创·实战派 S3 │   音频上下行+控制   │  xiaozhi-esp32-server-golang│
│ ·麦克风/喇叭   │                    │  + 安伴自研功能层            │
│ ·屏幕(脸)     │                    │   ·主动问候/提醒调度         │
│ ·摄像头       │                    │   ·留言收发                  │
│ ·小智固件     │                    │   ·家庭画像/记忆库 ★         │
└─────────────┘                    │   ·视觉触发处理              │
                                    └───────┬──────────┬─────────┘
                  ┌──────────────────────────┘          │ 调用
                  │ HTTP/WebSocket                       ↓
          ┌───────┴────────┐              ┌──────────────────────────┐
          │ 子女端 Web App   │              │ 云 API: ASR / LLM / TTS    │
          │ ·看状态/发留言    │              │ (火山/阿里等，含视觉VLM)     │
          │ ·触发问候/提醒    │              └──────────────────────────┘
          │ ·编辑家庭画像     │
          │ ·看一眼/视频通话  │
          └────────────────┘
```

★ 记忆库是新增模块。深读已证实 xiaozhi **内置 mem0 / memobase / memos**（→ 倾向路线 A 复用）；W1 周五正式拍板，见 [decisions/](./decisions/)。

**三句话读懂这张图：**
- 设备只负责"嘴+耳+眼"：录音、放音、显示、拍照，本身不思考。
- 后端是"大脑中转站"：接设备音频转给云 API 做识别/思考/合成，再把语音发回设备；安伴所有特色功能（主动调度、提醒、留言、记忆、视觉触发）都加在这一层。
- 子女端是"一个网页"：连后端，看状态、发留言、触发问候/提醒、编辑画像。

## 5. 技术选型

| 部分 | 选型 | 说明 |
|---|---|---|
| 设备端固件 | `xiaozhi-esp32`（烧 `lichuang-dev` 固件） | 不改或尽量少改，复用其录音/联网/播放/唤醒/打断 |
| 硬件 | 立创·实战派 ESP32-S3 复刻升级版 ×2 | 带电池、120°摄像头、双麦+AEC；一主一备 |
| 后端 | `xiaozhi-esp32-server-golang` | 主线，安伴功能加在其上 |
| 语音三件套 | 云 API（火山引擎/阿里等） | ASR+LLM+TTS 直接接，Demo 阶段不自建 |
| 视觉 | 云端 VLM（视觉大模型 API） | 设备拍照上传，云端理解 |
| **记忆库** | **倾向路线 A 复用**（深读已证实 xiaozhi 内置 mem0 / memobase / memos）；W1 周五正式拍板 | **详见 [decisions/<YYYY-MM-DD>-memory-module.md](./decisions/)**（W1 周五前定稿）。深读依据 [全景 §5](./specs/2026-05-29-xiaozhi-full-architecture-map.md) |
| 子女端 | Web 网页（手机浏览器可开） | 不做原生 App；登录用简化访问码 |
| 部署 | 云服务器 server A | 现成 Linux 服务器 |

> 术语：**ASR** 语音转文字 / **LLM** 大语言模型 / **TTS** 文字转语音 / **VLM** 看图说话的视觉大模型 / **WebSocket** 设备和服务器间持续双向的实时连接 / **API key** 调用云服务的钥匙 / **pgvector** PostgreSQL 的向量扩展，用于做语义相似度检索。

## 7. 风险与纪律

### 7.1 执行铁律（来自此前讨论，3 周内不变）

- **先通主链路再加功能**：基础语音对话（设备 ↔ 后端 ↔ 云 API）没跑通前，不要写任何安伴特色功能，否则出问题分不清是硬件/固件/网络/后端/API 哪一层。
- **能买就买**：语音能力一律接 API，Demo 期不自建模型。
- **两块板**：一主一备，刷砖/坏口能立刻换。
- **48 小时闸门**：真板子到手后，若 `xiaozhi-esp32 + golang 后端` 在 48 小时内跑不通基础语音对话，立刻降级到更易跑通的 `xiaozhi-server-go` 或原版 `xiaozhi-esp32-server`，不恋战。
- **隐私暂挂**：完整对话记录按"开发期功能"做，隐私/告知机制留到产品化阶段，但 PPT 里要诚实标注"未来需处理"。详见 §8。

### 7.2 风险/回退树（6 类 13 行）

| 类 | 触发 | 立即动作 | 后果 | 终极兜底 |
|---|---|---|---|---|
| 硬件-板砖 | W1 D1 烧固件无反应 | 换备用板（已买 2 块） | 延 0.5 天 | 两块都挂 → 借/买 ESP32 公版，砍 #5#6#7 |
| 硬件-摄像头 | W1 D4 拍照不通 | 视觉触发降到加分 | 视觉弱化 | 路演切兜底视频 |
| 网络-server A | 任意周挂 | 本地启后端 + 手机热点 | UI 略差 | 备用笔记本 + HDMI |
| 网络-API 限流 | 任意时刻 | 切备用 key/账号；降模型 | 延 0.5 天 | 没账号 → 切兜底视频 |
| 后端-48h 闸门 | W1 D3 跑不通 | 降级 `xiaozhi-server-go` / 原版 `xiaozhi-esp32-server` | 中级风险 | 按降级后路径继续走 |
| 后端-集成不稳 | W2 任意时刻反复失败 | 按优先级关闭安伴特色：记忆 → 视觉 → 提醒 → 留言 | 必演减少 | 必演 < 5 → 当晚补录兜底视频 |
| 记忆-决策超期 | W1 周五前未定路线 | 强制 Level 3（prompt 硬编码） | 记忆变兜底 | 现场用预写画像 prompt（观众看不出） |
| 记忆-集成不稳 | W2 中期不稳 | Level 1 → 2 → 3 顺序退（见下） | 功能弱化 | 同上 |
| 视觉-整条不稳 | W2 末仍不稳 | 视觉触发从必演移到加分 | 必演剩 6 | 切兜底视频 |
| 现场-投影 | 当天 | 手机投屏 / 备机 | 0 | 自带笔记本 + HDMI |
| 现场-网络 | 当天 | 手机热点 + 备机热点 | 0 | 关键句切兜底视频 |
| 现场-没电 | 当天 | 备板已充满；带充电线 | 0 | — |
| 现场-噪音 | 当天 | 凑近说话；备耳麦 | 小幅 | 关键句切兜底视频 |

**W2 反复失败时关闭安伴特色的优先级逻辑**：记忆是隐形的（关掉观众看不出来）→ 视觉是高光②的替代触发器之一 → 提醒是高光②的内容载体 → 留言是高光③的本体绝不能关。

### 7.3 记忆模块降级 4 级（与实现路径解耦）

```
Level 0（正常）：画像注入 + 对话沉淀 + 向量召回
        ↓ 召回不稳
Level 1：画像注入 + 对话沉淀，召回退到 LIKE 关键词
        ↓ 沉淀写入不稳
Level 2：仅画像注入，对话不沉淀
        ↓ 整条跑不通
Level 3：路演现场 prompt 硬编码画像，记忆功能切兜底视频
```

每一级都能演 #5 的高光时刻 1（来自画像的召回）。只有 Level 0 能完整演高光时刻 2（来自当前会话沉淀的召回）。

### 7.4 W1 周五双重 Checkpoint

设一个明确的"W1 周五评估"，两件事一起判：① 48 小时闸门是否已过 ② 记忆模块路线是否已定（A/B/C 之一）。

```
都通过           → W2 进入"加功能"模式（按三周计划矩阵 W2 列执行）
闸门没过         → 已降级到 xiaozhi-server-go/原版，W2 仍按矩阵执行只是后端换了
记忆决策未定     → 强制 Level 3，W2 跳过记忆集成，挪时间到视觉/子女端
两个都没过       → 进入"砍范围"模式：必演压到 5 条（#1+#2+#3+#4+#6），其余切兜底视频
```

这条规则白纸黑字写在这里，避免到了 W1 周五还在犹豫。
